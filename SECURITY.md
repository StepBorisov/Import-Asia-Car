# Безопасность — ZAVOZ

Документ описывает реализованные меры защиты и шаги, которые нужно выполнить
перед продакшн-деплоем.

## Реализовано в коде

| # | Требование | Как реализовано |
|---|------------|-----------------|
| 1 | Секреты не во фронтенде | Telegram token, Supabase service role key и прочие ключи используются только в `server/` и читаются из `process.env`. Во фронтенд-бандл ничего секретного не попадает. |
| 2 | Переменные окружения для секретов | Все секреты — через `.env` (локально) и Environment на Render. Шаблон — `.env.example`. |
| 3 | `.env` не в git | `.gitignore` исключает `.env`, `.env.*`, ключи и `node_modules`. |
| 4 | Серверная валидация | `server/validate.js` — единственный источник истины: проверка обязательных полей, хотя бы одного контакта, enum-значений (страна/тип), длины полей. Фронтенд-валидация — только для UX. |
| 5 | Санитизация ввода | Удаление control-символов, схлопывание пробелов, жёсткие лимиты длины; числовые поля — только цифры. |
| 6 | Защита от XSS | Динамический вывод во фронтенде экранируется (`esc()` в `shared.js`). В Telegram-сообщении — HTML-escape (`server/telegram.js`). Заголовок `X-Content-Type-Options: nosniff`. |
| 7 | Защита от SQL-инъекций | Запись через параметризованный клиент Supabase (`.insert(...)`), без конкатенации SQL. |
| 8 | Rate limiting / антиспам | `server/rateLimit.js` — лимит заявок на IP за окно (по умолчанию 5 / 10 мин), ответ `429` с `Retry-After`. |
| 9 | Honeypot | Скрытое поле `website`: если заполнено (бот) — заявка тихо отклоняется. |
| 10 | Защита от дублей | Серверная идемпотентность по хешу ключевых полей в течение 60 сек + блокировка повторной отправки на фронтенде. |
| 11 | Row Level Security | `supabase/schema.sql` включает RLS **без** разрешающих политик для anon/authenticated → публичный ключ не имеет доступа к таблице. |
| 12 | Ограничение размера запроса | `express.json({ limit: '32kb' })` — защита от больших payload'ов. |
| 13 | Security-заголовки | CSP, `X-Frame-Options: DENY`, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, а также HSTS в production. |
| 14 | CSP | `default-src 'self'`; скрипты/стили/шрифты/изображения ограничены собственным origin (+ `data:` для картинок), `frame-ancestors 'none'`. Внешние скрипты и подключения запрещены. |
| 15 | CORS | По умолчанию — same-origin. Дополнительные origin'ы — только явно через `ALLOWED_ORIGINS`. |
| 16 | Отказоустойчивость каналов | Сбой Telegram не теряет заявку (сохраняется в БД) и наоборот; при отказе обоих — честный ответ об ошибке пользователю, а не ложный «успех». |
| 17 | Отсутствие утечки логики | Пользователю не раскрываются внутренние ошибки БД/Telegram — только нейтральные сообщения; детали пишутся в серверные логи. |
| 18 | `x-powered-by` отключён | `app.disable('x-powered-by')`. |

## Проверить перед деплоем

- [ ] Заданы `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`, `SUPABASE_URL`,
      `SUPABASE_SERVICE_ROLE_KEY` в панели Render (не в коде).
- [ ] Выполнен `supabase/schema.sql`; RLS включён (проверьте, что для таблицы
      `requests` **нет** разрешающих политик для `anon`).
- [ ] `.env` **не** закоммичен (проверьте `git status` и историю).
- [ ] Сайт открывается только по HTTPS (Render обеспечивает автоматически).
- [ ] При необходимости обновлён `ALLOWED_ORIGINS` под реальный домен.
- [ ] Проверена работа формы: успешная отправка приходит в Telegram и в таблицу
      Supabase.
- [ ] Проверен rate limit (после `RATE_LIMIT_MAX` заявок с одного IP — `429`).

## Модель угроз (кратко)

- **Спам/абьюз формы** → rate limiting + honeypot + дедупликация + лимит размера.
- **XSS** → экранирование при выводе (фронт и Telegram) + строгий CSP.
- **Инъекции** → параметризованные запросы Supabase, валидация enum/длины/типов.
- **Утечка секретов** → секреты только на сервере, `.env` в `.gitignore`,
  service role key недоступен клиенту.
- **Несанкционированный доступ к данным** → RLS с default-deny, доступ к таблице
  только через серверный service role key.

## Остаточные риски / рекомендации

- In-memory rate limit и дедупликация работают в пределах одного инстанса. При
  масштабировании — вынести в общий store (Redis/Upstash).
- Для публичных высоконагруженных форм рассмотреть добавление CAPTCHA.
- Регулярно ротируйте ключи Supabase и токен бота; при подозрении на утечку —
  перевыпустите немедленно.
